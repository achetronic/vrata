# Technical Decisions

Technical decisions made by the project owner. Any AI tool working on this repository
**must respect these decisions** and not revert them without explicit approval.

---

## xDS for live Envoy configuration (no static files, no reloads)

**Date**: 2026-03-15
**Status**: Decided

Rutoso pushes configuration to Envoy proxies via the xDS protocol (gRPC streaming) using
the official `go-control-plane` library. It does not generate static Envoy YAML files and
does not rely on Envoy's `--config-path` or hot-reload mechanisms.

**Reasoning**: Target Envoys handle thousands of requests per minute. Any mechanism that
requires a reload introduces a risk of dropped connections or downtime. xDS allows
configuration changes — new routes, updated backends, traffic splits — to be applied
instantaneously without touching the proxy process. This is the primary reason Envoy
supports xDS at all.

**Do not**: Generate static Envoy config files as the primary delivery mechanism.
Do not use Envoy's file-based config reload (`kill -SIGHUP`) even as a fallback.

## net/http as the REST API router (no external framework)

**Date**: 2026-03-15
**Status**: Decided

The REST API is built on Go's standard library `net/http`. No external HTTP framework
(Gin, Echo, Chi, Fiber, etc.) is used unless a concrete, documented justification is
provided and approved.

**Reasoning**: Rutoso's API surface is well-defined and not complex enough to justify
a framework dependency. `net/http` is stable, has zero transitive dependencies, and is
sufficient for routing, middleware chaining, and JSON handling at this scale. Minimising
external dependencies reduces supply-chain risk and simplifies long-term maintenance.

**Do not**: Add Gin, Echo, or any other HTTP router/framework without an explicit decision
to do so. Do not work around `net/http` limitations with one-off hacks instead of
reconsidering the decision properly.

## slog for structured logging

**Date**: 2026-03-15
**Status**: Decided

`log/slog` from the Go standard library is the only logging library used in Rutoso.
It supports both human-readable (text) and JSON output formats, controlled by config.

**Reasoning**: slog is lightweight, has no external dependencies, and covers all logging
needs for this project. Adding a third-party logger (zerolog, zap, logrus) would be
adding dependency weight without a meaningful capability gain.

**Do not**: Import zerolog, zap, logrus, or any other third-party logging library.
Do not use `fmt.Println` or `log.Printf` for application logging.

## swag-go v2 for OpenAPI 3.1 documentation

**Date**: 2026-03-15
**Status**: Decided

API documentation is generated via swag-go v2 (`github.com/swaggo/swag/v2`), which
produces an OpenAPI 3.1 spec from annotations in handler source code. The generated
output lives in `docs/` and is committed to the repository.

**Reasoning**: OpenAPI 3.1 is the current standard and has better support for JSON Schema
constructs needed for complex route/match rule definitions. swag-go v2 is the only version
that targets OpenAPI 3.1; v1 only produces OpenAPI 2.0 (Swagger).

**Do not**: Use swag-go v1 annotations or syntax. Do not edit the `docs/` directory
manually — it is fully generated by `make docs`. Do not replace swag-go with another
spec generator without an explicit decision.

## go-control-plane as the xDS implementation

**Date**: 2026-03-15
**Status**: Decided

The xDS gRPC server is implemented using `github.com/envoyproxy/go-control-plane`,
the official Envoy xDS library for Go maintained by the Envoy project itself.

**Reasoning**: Using the official library ensures protocol correctness, compatibility
with current and future Envoy versions, and access to maintained snapshot cache
primitives. Third-party or hand-rolled xDS implementations introduce significant
risk of subtle protocol violations.

**Do not**: Implement xDS from scratch using raw gRPC protobuf definitions.
Do not use an unofficial or community fork of go-control-plane.

## os.ExpandEnv applied before YAML unmarshalling

**Date**: 2026-03-15
**Status**: Decided

When loading `config.yaml`, the raw file bytes are passed through `os.ExpandEnv` before
being unmarshalled into the `Config` struct. This allows any string value in the config
to reference environment variables using `${VAR_NAME}` syntax.

**Reasoning**: Makes Rutoso trivially deployable in containerised environments (Kubernetes,
Docker Compose) where secrets and environment-specific values are injected as environment
variables without needing a separate templating tool.

**Do not**: Use a YAML templating library (gomplate, envsubst binary, etc.) as a
preprocessing step. Do not read environment variables directly inside components at
runtime — all config must flow through the `Config` struct loaded at startup.

## Storage backend: bbolt (embedded KV store, single-file)

**Date**: 2026-03-16
**Status**: Implemented

The persistent store implementation uses `go.etcd.io/bbolt` — an embedded, pure-Go
key/value database that stores all data in a single file (`rutoso.db` by default,
configurable via `--store-path`). Two buckets are used: `groups` and `routes`,
both storing JSON-encoded domain objects keyed by their UUID.

**Reasoning**: bbolt requires zero external processes, zero infrastructure, and produces
a single file that can be backed up with a plain `cp`. It supports concurrent reads (multiple
readers) with serialised writes, which matches Rutoso's workload. For HA deployments with
multiple replicas, the database file can be placed on shared storage (NFS, CSI volume)
or replicated with an external tool. This satisfies all stated requirements with minimum
operational complexity.

**Do not**: Switch to an external database (PostgreSQL, etcd, Redis) without an explicit
decision. Do not read or write the bbolt file from outside the `store/bolt` package.
Do not remove the `store.Store` interface — all components must remain storage-agnostic.
