# Technical Decisions

Technical decisions made by the project owner. Any AI tool working on this repository
**must respect these decisions** and not revert them without explicit approval.

---

## Live configuration via xDS ADS (no static files, no reloads)

**Date**: 2026-03-29
**Status**: Implemented (supersedes SSE approach from main branch)

Vrata pushes configuration to Envoy instances via xDS Aggregated Discovery
Service (ADS) over gRPC. Every store mutation triggers a full snapshot rebuild
and push. Envoy applies changes atomically without restarts.

**Reasoning**: xDS is Envoy's native configuration protocol. It provides typed,
versioned resource delivery with built-in consistency guarantees. Every store
change triggers a full rebuild — partial updates would introduce inconsistency
between resource types.

**Do not**: Generate static Envoy config files. Do not require Envoy restarts
for config changes. Do not implement partial xDS updates (EDS-only, CDS-only).
Always rebuild and push the full snapshot.

---

## Envoy as the data plane (replaces native Go proxy)

**Date**: 2026-03-29
**Status**: Implemented (supersedes native Go proxy from main branch)

Vrata is a pure control plane. The data plane is a fleet of Envoy instances
that connect to the xDS gRPC server and receive configuration dynamically.
The native Go proxy (`internal/proxy/`), SSE sync client (`internal/sync/`),
and Go middleware implementations have been removed.

**Reasoning**: Envoy provides battle-tested L7 proxying with native support
for TLS/mTLS, circuit breaking, outlier detection, retries, rate limiting,
CORS, JWT, ext_authz, and more. Reimplementing all of this in Go is
significant ongoing maintenance. Envoy's Go plugin system covers the gaps
(sticky sessions, inline CEL authz, XFCC injection).

**Do not**: Re-introduce a native Go proxy. Do not add `internal/proxy/`
or `internal/sync/` back. All traffic routing is done by Envoy.

## net/http as the REST API router (no external framework)

**Date**: 2026-03-15
**Status**: Decided

The REST API is built on Go's standard library `net/http`. No external HTTP framework
(Gin, Echo, Chi, Fiber, etc.) is used unless a concrete, documented justification is
provided and approved.

**Reasoning**: Vrata's API surface is well-defined and not complex enough to justify
a framework dependency. `net/http` is stable, has zero transitive dependencies, and is
sufficient for routing, middleware chaining, and JSON handling at this scale. Minimising
external dependencies reduces supply-chain risk and simplifies long-term maintenance.

**Do not**: Add Gin, Echo, or any other HTTP router/framework without an explicit decision
to do so. Do not work around `net/http` limitations with one-off hacks instead of
reconsidering the decision properly.

## slog for structured logging

**Date**: 2026-03-15
**Status**: Decided

`log/slog` from the Go standard library is the only logging library used in Vrata.
It supports both human-readable (text) and JSON output formats, controlled by config.

**Reasoning**: slog is lightweight, has no external dependencies, and covers all logging
needs for this project. Adding a third-party logger (zerolog, zap, logrus) would be
adding dependency weight without a meaningful capability gain.

**Do not**: Import zerolog, zap, logrus, or any other third-party logging library.
Do not use `fmt.Println` or `log.Printf` for application logging.

---

## swag-go v2 for OpenAPI 3.1 documentation

**Date**: 2026-03-15
**Status**: Decided

API documentation is generated via swag-go v2 (`github.com/swaggo/swag/v2`), which
produces an OpenAPI 3.1 spec from annotations in handler source code. The generated
output lives in `docs/` and is committed to the repository.

**Do not**: Use swag-go v1 annotations or syntax. Do not edit the `docs/` directory
manually — it is fully generated by `make docs`. Do not replace swag-go with another
spec generator without an explicit decision.

## os.ExpandEnv applied before YAML unmarshalling

**Date**: 2026-03-15
**Status**: Decided

When loading `config.yaml`, the raw file bytes are passed through `os.ExpandEnv` before
being unmarshalled into the `Config` struct. This allows any string value in the config
to reference environment variables using `${VAR_NAME}` syntax.

**Do not**: Use a YAML templating library. Do not read environment variables directly
inside components at runtime — all config must flow through the `Config` struct loaded
at startup.

## Storage backend: bbolt (embedded KV store, single-file)

**Date**: 2026-03-16
**Status**: Implemented

The persistent store implementation uses `go.etcd.io/bbolt` — an embedded, pure-Go
key/value database that stores all data in a single file. The path is derived from
`controlPlane.storePath` in config.yaml.

**Do not**: Switch to an external database without an explicit decision. Do not read
or write the bbolt file from outside the `store/bolt` package. Do not remove the
`store.Store` interface — all components must remain storage-agnostic.

---

## Routes as independent first-class entities (not embedded in groups)

**Date**: 2026-03-16
**Status**: Implemented

A `Route` is a standalone entity stored and managed independently of any `RouteGroup`.
A `RouteGroup` references routes by their IDs (`RouteIDs []string`) and adds group-level
matchers (`PathPrefix`, `Hostnames`, `Headers`) that are merged on top of each referenced
route's own match rules at xDS translation time.

**Do not**: Embed `Route` objects directly inside `RouteGroup`. Do not add a `GroupID` field
back to `Route`. Routes and groups must remain separate resources at both the API level
and the storage level.

---

## Filter as independent first-class entity (not embedded in Listener)

**Date**: 2026-03-16
**Status**: Implemented

A `Middleware` (filter) is a standalone entity stored and managed independently of any
`Listener`. The xDS translator looks up each referenced middleware and builds the
corresponding Envoy HTTP filter.

**Do not**: Embed filter configs directly inside `Listener`. Filters and Listeners must
remain separate resources at both API level and storage level.

---

## FilterOverrides on both Route and RouteGroup, with Route-wins merge semantics

**Date**: 2026-03-16
**Status**: Implemented

Both `Route` and `RouteGroup` carry a `MiddlewareOverrides map[string]MiddlewareOverride`
field, keyed by middleware ID. The merge order is: RouteGroup overrides are the base;
Route overrides win on key collision (more specific wins).

**Do not**: Invert the merge direction. Do not remove `MiddlewareOverrides` from
`RouteGroup`. Do not flat-replace: always merge (group base + route delta).

---

## Route action grouped under ForwardAction (not flat on Route)

**Date**: 2026-03-16
**Status**: Implemented

All forwarding behaviour (destinations, timeouts, retry, rewrite, mirror) lives inside
a `ForwardAction` struct nested under `Route.Forward`. The three action modes are:

- `Route.Forward *ForwardAction` — proxy to upstream(s)
- `Route.Redirect *RouteRedirect` — HTTP redirect to client
- `Route.DirectResponse *RouteDirectResponse` — fixed response, no upstream

**Do not**: Move `Timeouts`, `Retry`, `Rewrite`, or `Mirror` back to the `Route`
root level. Do not add a `type` discriminator field — the presence of the relevant
pointer is the discriminator.

---

## TLS termination modeled on Listener

**Date**: 2026-03-16
**Status**: Implemented

`model.Listener` carries an optional `TLS *ListenerTLS` field (cert/key paths, min/max
protocol version, mTLS client auth with CA file and mode). The xDS translator maps this
to an Envoy `DownstreamTlsContext` on the filter chain's transport socket.

**Do not**: Move TLS config to a separate entity. Do not remove `TLS` from the model.

---

## Destination as the upstream abstraction — DestinationRef replaces inline Backend

**Date**: 2026-03-16
**Status**: Implemented

Upstream backends are modeled as first-class `Destination` entities. Routes reference
destinations via `DestinationRef` (`destinationId`, `weight`). The old `Backend` struct
has been removed entirely.

**Do not**: Re-introduce an inline `Backend` struct on `Route`. `DestinationRef` must
reference a `Destination` by ID — never embed the destination config directly.

---

## Cluster type is auto-derived — no user-facing discovery field on Destination

**Date**: 2026-03-16
**Status**: Implemented

The Envoy cluster discovery type is derived automatically from the Destination fields:

- `options.discovery.type == "kubernetes"` → EDS
- `host` is a bare IPv4/IPv6 address → STATIC
- `host` is an FQDN → STRICT_DNS

The derivation logic lives in `xds/server.go:clusterTypeFor`.

**Do not**: Add a `discoveryType` or `clusterType` field directly on `Destination`.

---

## hashPolicy lives on EndpointBalancing in Destination, not on ForwardAction

**Date**: 2026-03-18
**Status**: Implemented

`HashPolicy` is a field of `EndpointBalancing` (inside `Destination.Options`).
The LB algorithm and its hash policy live together under the algorithm-specific
nested struct.

There are two levels of load balancing:

- **Level 1 (destination selection)**: configured on the Route via
  `ForwardAction.DestinationBalancing`. The xDS translator maps this to
  Envoy route-level `HashPolicy` entries.
- **Level 2 (endpoint selection within a destination)**: configured on the
  Destination via `Options.EndpointBalancing`. The xDS translator maps this
  to the Envoy cluster `LbPolicy` and ring hash/maglev config.

**Do not**: Move `HashPolicy` back to `ForwardAction` or `Route`. Do not mix
level-1 and level-2 concerns in the same struct.

---

## discovery on Destination is Vrata-internal (no namespace/service fields)

**Date**: 2026-03-16
**Status**: Implemented

`Destination.Options.Discovery` signals that endpoint discovery should be
delegated to the Kubernetes EndpointSlice watcher. The `host` field holds
the full service FQDN. No separate `namespace` or `serviceName` fields.

**Do not**: Add `namespace`, `serviceName`, or `serviceAccount` fields to
`Destination` or `Discovery`.

---

## Route has three mutually exclusive action modes

**Date**: 2026-03-16
**Status**: Implemented

A `Route` operates in exactly one mode: `forward`, `redirect`, or `directResponse`.
All three are translated to their Envoy equivalents (RouteAction, RedirectAction,
DirectResponseAction).

**Do not**: Add a `type` discriminator field. Do not silently ignore extra fields
if more than one mode is set.

---

## Route action fields use semantic naming, not implementation internals

**Date**: 2026-03-16
**Status**: Implemented

Route-level behaviour uses user-facing semantic names. The translation to Envoy
xDS field names lives entirely in `xds/server.go`. Model types never reference
Envoy implementation field names.

**Do not**: Use Envoy field names in the API. Model types must remain
proxy-agnostic.

---

## k8s EndpointSlice watcher is non-fatal and opt-in via kubeconfig

**Date**: 2026-03-16
**Status**: Implemented

The Kubernetes EndpointSlice watcher is started only when a valid k8s client
can be built. If neither in-cluster config nor kubeconfig is available, the
watcher is silently disabled.

**Do not**: Make the k8s client a required dependency. Do not crash if no
kubeconfig is found.

---

## k8s watcher triggers gateway Rebuild — no separate EDS update path

**Date**: 2026-03-16
**Status**: Implemented

When an EndpointSlice event fires, the watcher calls `gateway.Rebuild(ctx)` —
the same full xDS snapshot rebuild used for store events. There is no separate
EDS-only push path.

**Do not**: Implement a partial EDS update. Do not bypass `gateway.rebuild`.

---

## Header manipulation is a Middleware, not a Route/Group field

**Date**: 2026-03-16
**Status**: Implemented

Request/response header add/remove is modeled as a `Middleware` entity (type
`headers`). The xDS translator maps this to `envoy.filters.http.header_mutation`.

**Do not**: Add `requestHeadersToAdd`, `requestHeadersToRemove`, etc. as direct
fields on `Route` or `RouteGroup`.

---

## Versioned snapshots as the source of truth (REST API)

**Date**: 2026-03-17
**Status**: Implemented (API preserved, semantics differ from main branch)

The REST API still exposes `POST /snapshots` and `POST /snapshots/{id}/activate`.
However, in this branch the xDS push happens on every store change, not only
on snapshot activate. The snapshot endpoints exist for auditing and rollback
but are not the gating mechanism for config delivery.

**Do not**: Remove the snapshot endpoints from the API.

## CEL expressions as complementary route matcher

**Date**: 2026-03-17
**Status**: Model preserved, xDS translation pending

`MatchRule` carries an optional `CEL` field. In the native proxy this was
compiled and evaluated per-request. In the Envoy branch, CEL route matching
is not yet translated to xDS (pending: RBAC filter or Go plugin approach).

**Do not**: Remove CEL from the model. The translation mechanism is pending
but the API contract must remain stable.

**Date**: 2026-03-17

The external processor middleware uses a Vrata-native gRPC service definition.
The xDS translator does not yet emit `envoy.filters.http.ext_proc` config.

The `BackendRef` type was renamed to `DestinationRef` and the `Backends`

## Two-level load balancing: DestinationBalancing + EndpointBalancing

**Date**: 2026-03-18
**Status**: Implemented

### Level 1 — Destination selection (Route)

`ForwardAction.DestinationBalancing` controls which Destination receives each
request. Translated to Envoy route-level hash policy:

- `WEIGHTED_RANDOM` (default) — no hash policy
- `WEIGHTED_CONSISTENT_HASH` — cookie hash policy on route
- `STICKY` — cookie hash policy + Go plugin for Redis lookup

### Level 2 — Endpoint selection (Destination)

`Destination.Options.EndpointBalancing` controls which endpoint within a
Destination receives each request. Translated to Envoy cluster `LbPolicy`:

- `ROUND_ROBIN`, `RANDOM`, `LEAST_REQUEST` — direct mapping
- `RING_HASH`, `MAGLEV` — direct mapping with ring size / table size config

**Do not**: Mix level-1 and level-2 configuration in the same struct.

---

## HA control plane: Raft consensus with transparent write-forwarding

**Date**: 2026-03-18
**Status**: Implemented

Multiple control plane replicas share configuration via embedded Raft consensus.
Write-forwarding from followers to leader. Every node serves xDS reads from
local bbolt.

**Do not**: Use eventual consistency for config replication. Do not require
Envoy to know which node is the Raft leader. Do not run more than 5 voting nodes.

---

## onError fallback actions on Route

**Date**: 2026-03-19
**Status**: Model preserved, xDS translation not applicable

`Route.OnError` is preserved in the model for API compatibility. Envoy handles
error fallbacks differently (via retry policy, circuit breaking, and outlier
detection). The onError rules are not translated to xDS.

**Do not**: Remove `OnError` from the model.

---

## Config sections named after their mode: controlPlane and proxy

**Date**: 2026-03-19
**Status**: Implemented

- `controlPlane: { address, xdsAddress, storePath, raft }` — control plane config
- `proxy: { controlPlaneUrl, reconnectInterval }` — proxy mode config (unused in this branch, Envoy replaces it)

**Do not**: Re-introduce `server:` or `cluster:` as top-level sections.

## Timeout naming convention: semantic names mapped to Envoy fields

**Date**: 2026-03-19
**Status**: Implemented

All configurable timeouts use semantic names. The xDS translator maps them
to Envoy proto fields:

### Destination timeouts → Envoy Cluster

| Semantic name | Envoy field |
|---|---|
| `connect` | `Cluster.ConnectTimeout` |

### Route timeouts → Envoy Route

| Semantic name | Envoy field |
|---|---|
| `forward.timeouts.request` | `RouteAction.Timeout` |

### Middleware timeouts → Envoy filter config

| Middleware | Field | Envoy mapping |
|---|---|---|
| ExtAuthz | `decisionTimeout` | `GrpcService.Timeout` / `HttpService.ServerUri.Timeout` |
| JWT | `jwksRetrievalTimeout` | `RemoteJwks.HttpUri.Timeout` |

Listener timeouts (`clientHeader`, `clientRequest`, `clientResponse`,
`idleBetweenRequests`) exist in the model but are not yet mapped to Envoy
HCM settings. Envoy has its own listener-level timeouts configured separately.

**Do not**: Use Envoy field names in the API.
